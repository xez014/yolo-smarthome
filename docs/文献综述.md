# 毕业设计文献综述

**题目：** 基于YOLO的智能家居视觉感知与历史物品检索系统文献综述

## 摘要

随着智能摄像头、家庭宽带网络与物联网终端的普及，家居视频数据正在从“被动录像”逐步转向“主动理解”。本文围绕“基于YOLO的智能家居视觉感知与历史物品检索系统”的设计目标，对目标检测算法、室内复杂场景检测、多目标跟踪、浏览器媒体采集与 WebSocket 推流等相关研究和工程资料进行梳理。综述表明，YOLO 系列检测器具备实时检测优势，YOLOv8 与 YOLO11 的工程化工具链适合毕业设计规模的模型训练、验证和部署；ByteTrack 可在不引入额外 ReID 分支的情况下改善遮挡场景下的轨迹连续性；浏览器 `getUserMedia` 与 WebSocket 二进制传输可为家庭内网摄像头难以公网访问的问题提供一种工程替代方案。上述研究和技术资料为本系统构建 SmartHome-COCO 数据集、训练 YOLO11s 模型、设计前后端分离系统和实现历史物品检索提供了依据。

**关键词：** 智能家居；目标检测；YOLO11；ByteTrack；WebSocket；历史物品检索

## 1 引言

传统智能家居摄像头通常提供实时预览、录像保存和事件回放等功能，但较少把画面中的物品类别、位置、时间和快照转化为可查询的数据。当系统能够把视频中的物品检测结果结构化落库后，用户即可通过“笔记本最后出现在哪里”“手机最近一次出现时间”等方式进行检索。该需求既依赖目标检测模型，也依赖视频接入、数据库持久化和前端可视化等工程环节。

本课题以室内物品的实时检测、连续跟踪与历史记录检索为核心目标，采用前后端分离的 B/S 架构：后端使用 FastAPI、OpenCV、Ultralytics 和 SQLAlchemy 完成推理与数据持久化；前端使用 Vue 3、Element Plus 和 ECharts 完成实时监控、历史检索和统计展示；视频接入同时支持服务端网络流拉取与浏览器客户端 WebSocket 推流。围绕这些技术方向，本文从目标检测、室内场景适应、多目标跟踪、流媒体通信和系统集成五个方面进行文献综述。

## 2 目标检测算法与 YOLO 工程体系

目标检测任务需要同时完成目标类别判断和边界框定位。Faster R-CNN 通过区域建议网络提升了两阶段检测器的效率，在深度学习目标检测发展中具有代表性意义。但两阶段检测器通常流程较长，不适合对连续视频流进行低延迟分析。Redmon 等提出的 YOLO 将目标检测建模为单阶段回归问题，使模型能够在一次前向传播中直接输出边界框和类别概率，从而显著提升实时性。

近年来，Ultralytics YOLO 系列在训练、验证、推理、导出和跟踪接口方面形成了较完整的工程工具链。YOLOv8 官方文档介绍了其 anchor-free 检测头和多规模模型设计，使开发者可以在精度和速度之间进行选择[1]。YOLO11 进一步提供 n、s、m、l、x 等不同规模模型，官方模型卡列出了各模型在 COCO 检测任务上的参数量、计算量和精度指标[2]。对于本课题而言，YOLO11 的价值不仅在于模型结构本身，还在于其与 Python API、训练脚本、验证脚本和跟踪器配置的良好集成，便于在 SmartHome-COCO 子集上完成迁移训练并接入后端推理服务。

## 3 室内复杂场景检测研究

家居环境与道路、工业检测等场景不同，常见问题包括光照不稳定、目标尺度跨度大、桌面物品遮挡频繁、小件物品易漏检等。Zhao 等在智能室内购物场景中改进 YOLOv8n，引入 C2f-ParNet、CARAFE 和 WIoUv3 等模块，以提升复杂室内环境中小中型目标和遮挡目标的检测效果[3]。虽然该研究场景并非家庭物品检索，但其关注点与本课题中的室内小目标、遮挡和轻量模型部署具有较强关联。

Pokuciński 与 Mrozek 研究了 YOLOv5 在室内等距矩形全景图像中的目标检测效果，指出室内全景图像虽然覆盖范围更大，但畸变、高分辨率和光照不连续会提高检测难度[4]。这说明室内视觉感知不能只依赖模型套用，还需要结合采集视角、输入分辨率、数据分布和部署方式进行综合设计。

国内相关研究也从小目标检测、复杂环境鲁棒性和模型结构改进等角度提供了参考。郭磊等围绕改进 YOLOv5 的小目标检测展开研究，强调特征表达和小目标召回能力的重要性[9]；章程军等在车辆目标检测任务中通过结构改进和注意力机制增强复杂环境下的检测效果[10]；赵畅、王选的学位论文分别从 YOLOv5 改进和复杂环境目标检测角度讨论了网络结构、参数优化、特征融合和注意力机制等问题[11-12]。这些研究虽不直接面向智能家居，但其关于小目标、遮挡、光照变化和复杂背景的处理思路，可为本课题的模型选型和后续优化提供参考。

## 4 多目标跟踪与历史记录去重

单帧目标检测只能判断当前帧中的物品类别和位置，不能稳定描述同一物品在连续帧中的出现过程。如果将每帧检测结果全部写入数据库，会造成大量重复记录；如果完全忽略时序信息，又难以判断目标是否稳定出现。因此，多目标跟踪和业务层去重是构建历史物品检索系统的重要环节。

Deep SORT 在 SORT 的基础上加入外观特征度量，通过 ReID 特征缓解遮挡带来的身份切换问题。不过，额外特征提取会增加推理开销。ByteTrack 采用“关联每一个检测框”的思路，将检测结果分为高置信度和低置信度两组，先用高置信度框完成主要匹配，再利用低置信度框恢复被遮挡或模糊的轨迹[5]。这种方式不依赖额外 ReID 模型，适合与实时检测器组合。

Ultralytics 跟踪模式文档提供了 `bytetrack.yaml` 作为内置跟踪器配置，并支持通过 `model.track(..., persist=True)` 在视频流场景中保持轨迹状态[6]。本课题后端正是基于该接口维护 `track_id`，并在应用层增加连续帧确认、数据库写入间隔和快照保存间隔，从而减少单帧误检、光照波动和重复写库对历史记录质量的影响。

## 5 浏览器媒体采集与 WebSocket 推流

家庭摄像头通常处于内网环境，公网服务器直接拉取 RTSP 视频流会受到 NAT、路由器端口映射和设备权限限制。浏览器端媒体采集为此提供了一种替代路径。MDN 文档指出，`getUserMedia()` 可在用户授权后获取摄像头或麦克风媒体流，但该 API 需要安全上下文，即 HTTPS 或 localhost 环境[7]。因此，若系统部署到公网并调用本地摄像头，前端站点需要通过 HTTPS 提供服务。

WebSocket 为浏览器和服务端之间提供长连接通信能力，能够发送文本或二进制数据[8]。本课题的客户端推流模式使用 `getUserMedia` 或本地视频文件作为输入源，将视频帧绘制到 Canvas 后编码为 JPEG Blob，再通过 WebSocket 二进制消息发送给 FastAPI 后端。后端完成单帧 YOLO 推理后返回 Base64 标注图和结构化检测结果。该方案属于浏览器媒体采集与 WebSocket 帧级推理链路，优点是实现简单、便于与现有 HTTP 鉴权和后端推理接口整合；不足是单帧推理天然缺少跨帧 TrackID，需要通过写库节流和快照间隔控制降低数据冗余。

## 6 数据集与系统集成依据

COCO 数据集提供了日常场景中大量实例级目标检测标注，是目标检测模型预训练和迁移学习中常用的数据来源。本课题基于 COCO 2017 筛选出与家居场景关系较强的 20 类物品，构建 SmartHome-COCO 子集，并转换为 Ultralytics YOLO 训练格式。该做法能够在毕业设计周期内降低人工标注成本，同时保留对椅子、沙发、电视、笔记本电脑、遥控器、手机、书籍等常见室内目标的检测能力。

从系统集成角度看，相关文献和工程资料分别支撑了本课题的不同层次：Faster R-CNN 和 YOLO 说明目标检测算法的发展脉络；Ultralytics 文档支撑模型训练、推理和跟踪接口选择[1-2][6]；室内检测研究支撑对光照、遮挡和小目标问题的分析[3-4][9-12]；ByteTrack 支撑视频目标跟踪和轨迹连续性的讨论[5]；MDN 文档支撑浏览器摄像头采集和 WebSocket 推流的工程可行性[7-8]；COCO 数据集支撑数据集构建方法。

## 7 综合评述

综合上述研究可以看出，YOLO 系列模型适合实时视觉感知场景，YOLO11 与 Ultralytics 工具链能够降低模型训练和部署成本；室内复杂场景检测需要关注小目标、遮挡、光照变化和数据分布偏差；ByteTrack 能够在不显著增加推理负担的前提下改善视频目标轨迹连续性；浏览器媒体采集与 WebSocket 推流能够绕开家庭摄像头公网可达性问题，但需要注意 HTTPS 约束、推流帧率、单帧无状态推理和数据库写入节流。

因此，本课题将轻量级目标检测、多目标跟踪、浏览器帧级推流、关系型数据库持久化和前后端可视化集成为一套智能家居视觉记忆系统，具有明确的工程实践意义。与已有研究相比，本课题的重点不在于提出新的检测网络结构，而在于把目标检测结果转化为可检索的结构化时空数据，并围绕家庭网络环境和毕业设计可部署性完成完整系统实现。
