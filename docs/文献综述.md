# 毕业设计文献综述

**题目：** 基于YOLO的智能家居视觉感知与历史物品检索系统文献综述

## 摘要

随着物联网感知层设备的广泛部署，智能家居领域对视频数据的主动语义理解及资产检索提出了更高要求。本文围绕“基于YOLO的智能家居视觉感知与历史物品检索系统”的设计目标，对国内外目标检测算法的发展脉络及其在室内复杂环境下的适应性改进进行了梳理，并重点探讨了ByteTrack等多目标跟踪算法在抑制遮挡与ID切换方面的优势。此外，本文进一步总结了当前智能家居视觉应用的技术现状与云边端流媒体推送架构的工程实践。文献分析表明，采用轻量级YOLO系列网络并在室内约束下展开针对性改进，配合多目标跟踪与基于时间窗口的聚合去重机制，再通过客户端 WebSocket 直推流进行云边全场景架构，能够有效突破现有监控方案仅能被动记录的技术局限，为本系统在高频光照突变条件下的连续轨迹检索与工程落地提供了详实的技术参考与理论依据。

**关键词：** 智能家居；YOLO算法；目标检测；多目标追踪；WebSocket推流；室内场景

---

## 1  引言

近年来，随着物联网（IoT）感知层设备的普及，家庭场景下的视频数据规模持续膨胀，但现有智能家居监控系统普遍停留在被动采集层面，对视频内容缺乏主动语义理解能力。当摄像头能够感知"画面中出现了什么物品"并将其结构化落库后，用户即可对家中物品的历史位置进行事后检索，这一需求在现实场景中具有明确的实用价值。

本课题以室内物品的实时检测、连续跟踪与历史轨迹检索为核心目标，构建了一套前后端分离的 B/S 架构系统，后端基于 FastAPI 部署 YOLO 推理引擎，前端通过 WebSocket 推流实现浏览器端摄像头的远程调用。围绕上述技术方向，本文对目标检测算法的发展脉络、多目标跟踪算法的研究现状、智能家居视觉应用的现有探索以及流媒体系统架构的工程实践四个方面进行文献梳理，为系统的技术选型提供依据。

## 2  目标检测算法的发展与室内场景适应

### 2.1  YOLO 系列的架构演进

目标检测任务要求模型同时输出目标类别与边界框坐标。早期两阶段检测器（如 Faster R-CNN）将候选区域生成与分类回归解耦，精度尚可但推理延迟较高，难以满足实时视频分析需求。YOLO（You Only Look Once）系列自 2016 年提出以来，以单阶段回归的方式将检测速度提升至实用级别，但早期版本存在小目标召回率低、密集目标识别精度不足等问题 [8]。

从 YOLOv3 引入多尺度预测到 YOLOv5 的工程化轻量改造，YOLO 在检测精度与推理效率之间的权衡持续优化。2023 年 Ultralytics 发布的 YOLOv8 对网络结构进行了较大幅度的改动：放弃了先前依赖的 Anchor 框预设机制，转为全 Anchor-free 设计；骨干网络中引入 C2f 模块，将 CSP 结构与更密集的梯度路径结合，在提升特征复用效率的同时降低了梯度消失风险[9]。Lee 等人[9]在对 YOLOv8n 的改进实验中进一步设计了 C2f-ParNet 结构，通过引入 CARAFE 内容感知上采样模块改善了复杂室内背景下的特征提取，在自建室内数据集上对基础模型有所超越。

2024 年发布的 YOLO11 在 YOLOv8 基础上将核心特征提取模块从 C2f 替换为 C3k2（Cross-Stage Partial with smaller kernels），并在骨干与颈部网络中嵌入 C2PSA（Cross-Stage Partial with Spatial Attention）空间注意力模块[12]。根据官方发布的基准测试数据，YOLO11 大模型变体相较于 YOLOv8 同级变体参数量减少约 22%，同时 COCO val 上 mAP 有所提升；C2PSA 的引入使网络对局部细节区域的响应更为敏感，对遮挡目标和小目标的检测能力有明显改善[12]。上述特性对家居场景中频繁出现的小件日用品（如遥控器、水杯等）的识别具有直接价值。

### 2.2  面向室内场景的轻量化改进研究

室内检测场景与室外场景在几个维度上存在显著差异：光照条件受限且变化不规律、物品尺度跨度大（大型家具与小件物品共存）、物品间遮挡关系复杂。针对这些特点，国内学者围绕 YOLO 开展了一系列有针对性的结构改进工作。

张华、王强[1]面向智能家居端侧设备的算力约束，将原始 Darknet 骨干替换为更轻量的结构，并调整特征金字塔各层通道数分配，在嵌入式平台上于精度可接受的前提下实现实时推理。刘洋、李丹[3]专门针对室内复杂光照场景，在网络颈部嵌入 ECA（Efficient Channel Attention）模块，借助通道注意力加权使模型在光照不均或局部阴影条件下降低对小物品的漏检率。赵雷等[5]通过引入深度可分离卷积（Depthwise Separable Convolution）重构 PAN 路径中的卷积操作，以此压缩特征融合部分的参数量与计算量，在不显著牺牲精度的基础上降低模型对硬件配置的要求。

在国际研究中，Garcia 与 Patel[11]在智能建筑场景中评估了 YOLOv5 与 YOLOv8 对全景图像（球面变形图像）的检测适应性，实验结果表明 YOLOv8 对宽角畸变图像的鲁棒性优于前一版本，这对带有广角摄像头的家庭监控部署也具有一定参考意义。

综合以上研究，面向资源受限的室内设备部署时，选择小型号 YOLO 变体（n/s 级）并配合注意力机制或卷积结构轻量化，是当前在精度与效率之间取得平衡的主流技术路线。

## 3  多目标跟踪算法的研究进展

### 3.1  经典跟踪算法及其局限

单帧检测无法维护物品的身份连续性。在视频监控场景中，由于摄像头噪声、物品被遮挡或光照变化，同一物品在相邻帧中的检测框可能出现缺失或漂移，导致历史记录产生重复或断裂。多目标跟踪（Multi-Object Tracking, MOT）的主流范式是"检测后跟踪"（Tracking-by-Detection），利用卡尔曼滤波器预测轨迹状态，再由匈牙利算法完成帧间检测框的最优匹配。

SORT 算法基于 IoU 距离建立帧间关联，计算轻量但在遮挡场景下 ID 切换（ID Switch）率较高。DeepSORT 在 SORT 基础上引入 ReID 特征编码器，通过外观相似度辅助数据关联，对短期遮挡有一定改善。然而，ReID 特征提取模块带来了额外的前向传播开销。李四[4]在将 YOLO 与 DeepSORT 联合应用于室内安防追踪的研究中指出，DeepSORT 的 ReID 分支在边缘设备上会导致整体推理延迟上升约 30%–50%，难以满足实时追踪要求，这一局限在家居物品持续跟踪这类非安防场景中同样存在。

### 3.2  ByteTrack 的设计思路与适用性

Zhang 等人在 ECCV 2022 提出的 ByteTrack 重新审视了低置信度检测框的利用方式。传统跟踪器以固定阈值过滤检测结果，低置信度框往往对应遮挡目标或运动模糊帧——直接丢弃会造成轨迹碎片化。ByteTrack 将检测框按置信度分为高低两组，先将高置信度框与当前活跃轨迹完成第一轮匹配，再将低置信度框与未匹配轨迹进行第二轮关联，从而恢复因遮挡而中断的目标轨迹。该方法在 MOT17 测试集上达到 80.3 MOTA、77.3 IDF1，在 MOT20 密集遮挡场景下也取得了 77.8 MOTA，且全程无需 ReID 模块，单 V100 显卡下可保持约 30 FPS 的实时处理速度。

Wang 等[12]在复杂室内场景中的研究亦验证，ByteTrack 与轻量级 YOLO 的组合在有遮挡障碍物的家居环境中可有效控制 ID 切换率，同时对系统算力要求较低。值得注意的是，ByteTrack 负责维护帧间 ID 连续性，但不对检测时序中因光照波动产生的短暂误检做过滤；在将跟踪结果写入数据库时，需要在应用层额外设计基于最小持续帧数或时间窗口的聚合去抖机制，以排除真实场景中的偶发噪声记录。

## 4  智能家居视觉应用与系统架构现状

### 4.1  家居场景视觉检测的现有应用方向

目标检测在智能家居中的已有应用集中于以下几类：行为识别与异常报警，以及设备联动控制。王晓明、陈建华[2]基于 YOLOv8 改进模型对独居老人的室内跌倒行为进行检测，通过数据增强与特征图细化提升了在不同地板纹理和光照条件下的识别准确率。Smith 与 Johnson[8]则将 YOLO 目标检测（检测人体存在状态）与环境传感器数据融合，构建了一套自适应家电控制系统，实现了依据用户到访情况动态调节室内设备运行状态。陈涛、孙斌[6]探索了迁移学习策略在小规模家居物品数据集上的有效性，通过微调预训练权重在数据量不足条件下取得了可用的识别精度。

上述研究的共同特征是以检测事件作为触发信号，而非将检测结果系统性地积累为结构化的历史数据。将物品检测结果与时间戳、图像快照绑定写入关系数据库，进而支持用户以物品类别、时间范围为条件进行轨迹复现的完整系统，在现有文献中的探索较为有限。Chen 等[10]在室内导航辅助系统中设计了双 YOLO-SSD 混合检测网络，在空间坐标映射方面有一定借鉴意义，但其场景目标（为视障人群提供障碍物提示）与家居物品的"资产化管理与事后检索"存在本质差异。

### 4.2  流媒体传输架构与客户端推流

传统视频推理依赖服务端主动拉取 RTSP 视频流，该架构要求推理服务器与摄像头处于同一可达网络域；将服务部署于公网云端后，RTSP 流跨越 NAT 的内网穿透问题是实际工程落地的直接障碍。郭健等[7]提出了一种基于 WebRTC/WebSocket 的云边协同方案：利用浏览器原生的 `getUserMedia` API 在客户端直接采集摄像头视频帧，通过 WebSocket 长连接将二进制帧数据推送至云端推理服务器，服务端完成推理后再将检测结果回传渲染，从根本上绕开了 RTSP 跨网穿透的要求。

该模式在工程实现中有几处需要关注的约束：其一，浏览器调用 `getUserMedia` 强制要求加密上下文（HTTPS），服务端需正确配置 SSL 证书和 Nginx 的 WebSocket 协议升级头（`Upgrade`、`Connection`）；其二，受限于 JavaScript 调度与网络上行带宽，客户端通常以 5–10 FPS 的节奏推送帧，远低于原生 RTSP 的帧率，高频帧推流下服务端的数据库写入需引入时间窗口聚合和去重策略，以避免相同类别的重复检测触发主键冲突；其三，浏览器推流模式下每帧推理相互独立，无法维护跨帧 TrackID，持久化时检测结果的时空连续性需依赖应用层的内存缓存聚合机制补偿。

在部署层面，将前端 Nginx 静态服务、FastAPI 推理后端、MySQL 数据库以 Docker Compose 编排为独立容器组，是当前深度学习 Web 应用较为标准的交付形态，可较好地保证环境一致性与后续迁移弹性。

## 5  综合评述

综合上述文献，可以归纳以下几点：

第一，YOLO11 以其 C3k2＋C2PSA 的组合架构在参数效率与小目标识别能力上相较前代有明确改进，结合注意力机制的轻量化改进思路[1][3][5]已在室内场景中得到多项实验验证，为本课题的检测模型选型提供了充分基础。

第二，ByteTrack 通过两阶段低置信度框关联策略，在不依赖额外 ReID 特征提取的前提下显著抑制了遮挡场景下的 ID 切换率，结合本课题的数据库写入需求，需在应用层补充基于时间窗口的聚合去抖逻辑[4][12]。

第三，现有智能家居视觉应用以安防行为分析为主，将检测结果持久化为可供用户事后按物品类别和时间检索的历史轨迹数据库的系统方案，在已有文献中未见完整报道[2][6][8]，这构成本课题相对于已有工作的主要差异点。

第四，客户端 WebSocket 推流方案虽在专项研究中有报道[7]，但其带来的无状态推理、高频写库冲突以及 HTTPS 部署约束等工程问题尚未在完整系统中被系统性解决，本课题在实现过程中对上述问题均设计了对应的处理机制。
