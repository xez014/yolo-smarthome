# YOLO-SmartHome 项目改进建议

整理日期：2026-04-17

## 一、当前项目完成情况概览

本项目已经形成了较完整的端到端智能家居视觉感知系统，主要由数据处理、模型训练、后端推理服务、前端可视化界面和 Docker 部署配置组成。系统以 YOLO 目标检测模型为核心，面向家庭场景中的常见物品进行检测、记录和检索，已经具备本科毕业设计展示所需的完整闭环。

当前仓库主要结构如下：

- `data_pipeline/`：数据工程模块，包含 SmartHome-COCO 数据集配置文件，目标类别来自 COCO 中的 20 类室内常见物体。
- `model_training/`：训练与模型检查脚本，包含一个部署候选权重 `weights/best.pt`。
- `results-01/`：主要训练实验产物，包含 YOLO11s 训练权重、训练参数、训练日志和可视化样例。
- `backend/`：FastAPI 后端，负责模型加载、视频流读取、WebSocket 推理、检测记录入库、快照保存、统计接口和初始化向导接口。
- `frontend/`：Vue 3 前端，包含实时监控大屏、物品检索页、统计分析页和系统初始化页。
- `Dockerfile.backend`、`Dockerfile.frontend`、`docker-compose.yml`：用于生产环境容器化部署。

根据当前部署情况，项目已经部署到一台 IPv6-only 的德国云服务器上，并通过 Cloudflare WARP 获得 IPv4 出口；对外访问侧已经使用域名、SSL 证书和 CDN 加速。这说明系统已经具备公网展示、HTTPS 摄像头权限调用和远程访问能力。

## 二、核心功能梳理

当前系统已经实现的核心功能包括：

1. 实时物品检测：后端加载训练好的 YOLO 模型，对服务端摄像头、视频文件、RTSP/HTTP 流或浏览器推送帧进行检测。
2. 客户端推流推理：前端可以调用本地浏览器摄像头或本地视频文件，通过 WebSocket 将 JPEG 帧推送到云端后端，再返回带检测框的结果图。
3. 多目标跟踪和去重：服务端拉流模式使用 Ultralytics 的 `model.track` 和 ByteTrack 配置，结合连续帧计数与写库节流减少重复记录。
4. 历史检索：系统将物品类别、置信度、归一化位置、检测时间、快照路径等信息写入 MySQL，支持按物品类别查询最后出现位置和历史记录。
5. 数据统计：按天、小时和类别统计检测频次，并在前端通过 ECharts 展示。
6. 图形化初始化：首次部署后可通过前端填写 MySQL 连接信息，后端自动保存配置并创建数据表。
7. Docker 部署：前端由 Nginx 托管并代理后端 API/WebSocket，后端容器内运行 FastAPI 和 YOLO 推理服务。

从功能完整性看，项目已经超过“模型训练 + 简单演示”的层次，形成了可部署、可交互、可记录、可检索的应用系统。

## 三、YOLO 模型与训练情况

当前仓库中存在两组值得区分的权重：

| 位置 | 模型来源 | 文件大小 | 参数量 | 类别数 | 训练信息 |
| --- | --- | ---: | ---: | ---: | --- |
| `model_training/weights/best.pt` | YOLO11n | 约 15.22 MB | 约 2.59M | 20 | checkpoint epoch 为 78，best_fitness 约 0.43777 |
| `results-01/runs/train_smarthome_s/weights/best.pt` | YOLO11s | 约 54.36 MB | 约 9.44M | 20 | checkpoint epoch 为 65，best_fitness 约 0.48428 |

当前后端配置 `backend/config.py` 中实际加载的是：

```text
results-01/runs/train_smarthome_s/weights/best.pt
```

也就是说，生产推理当前使用的是 YOLO11s 版本，而不是 `model_training/weights/best.pt` 中较小的 YOLO11n 权重。

训练数据配置包含 20 类家庭场景常见物品：

```text
chair, couch, potted plant, bed, dining table, toilet, tv, laptop, mouse, remote,
keyboard, cell phone, microwave, oven, toaster, sink, refrigerator, book, clock, vase
```

`results-01/runs/train_smarthome_s/results.csv` 中记录到第 66 个 epoch。最后一轮也是当前日志中的最佳 mAP50 轮次，主要指标为：

| 指标 | 数值 |
| --- | ---: |
| Precision | 0.67572 |
| Recall | 0.61442 |
| mAP@0.5 | 0.65508 |
| mAP@0.5:0.95 | 0.48428 |

从曲线看，模型性能从前几轮到第 66 轮持续提升，最后 5 轮仍有小幅上升，说明模型基本可用，但仍有继续训练、补充真实家庭场景数据和做类别级优化的空间。

## 五、可行改进建议

### 1. 增加系统访问控制

当前项目已经部署在公网域名下，并支持摄像头推流、数据库初始化、检测启停、历史快照访问等能力。如果没有访问控制，任何能够访问域名的人理论上都可能操作系统或查看快照。

建议：

- 给前端增加登录页，后端使用简单的管理员账号密码或 JWT Token 鉴权。
- 对 `/api/system/init-db`、`/api/system/test-db`、`/api/video/start`、`/api/video/stop`、`/api/video/ws/push` 等关键接口增加鉴权。
- 如果希望快速落地，可以先使用 Cloudflare Access 或 Nginx Basic Auth 保护整个站点。
- 对 WebSocket 推流接口限制单连接大小、连接数量和帧率，防止被滥用导致 CPU/GPU 资源耗尽。

毕业设计价值：

这部分可以写入“系统安全性设计”或“公网部署安全加固”章节，使项目从演示系统更接近真实应用。

### 2. 敏感配置安全化

当前数据库连接信息通过初始化页面保存到 `backend/data/database_config.json`，并且该目录会被 Docker volume 持久化。虽然使用方便，但数据库密码以明文保存，存在泄露风险。

建议：

- 将数据库密码优先改为环境变量或 Docker Compose secret。
- `database_config.json` 只保存非敏感信息，密码从环境变量读取。
- 确认 `backend/data/` 不会被 Git 提交，生产备份时也需要单独保护。
- 后端日志不要输出完整数据库连接串。

毕业设计价值：

可以作为“配置管理与安全性”优化点，实施成本较低，但工程意义明显。

### 3. 补全模型评估报告

当前已有 `results.csv`，可以说明整体 Precision、Recall、mAP 指标，但缺少每个类别的 AP、混淆矩阵、PR 曲线和真实样例对比。对于论文或答辩来说，仅有总体指标还不够直观。

建议：

- 补全 `model_training/val.py`，对当前 `results-01/runs/train_smarthome_s/weights/best.pt` 运行验证集评估。
- 输出每类物品的 Precision、Recall、mAP50、mAP50-95。
- 保存混淆矩阵、PR 曲线、F1 曲线和验证集预测样例。
- 将 `model_training/inspect_model.py` 中旧的 `results/runs/train_smarthome` 路径更新为当前 `results-01/runs/train_smarthome_s`。

毕业设计价值：

这部分可以直接增强论文中的“模型训练与实验分析”章节，使模型效果更有证据链。

### 4. 使用真实家庭场景数据继续微调

当前数据来自 COCO 子集，类别适合室内场景，但 COCO 图片与实际家庭监控视角存在差异，例如俯视角、低光照、遮挡、小目标、画面压缩、运动模糊等。

建议：

- 从实际部署摄像头或本地视频中抽帧，人工标注少量真实家庭场景数据。
- 优先补充小物体和常丢失物体，如 remote、mouse、cell phone、book。
- 增加“钥匙、钱包、眼镜、耳机、充电器”等更符合“寻找物品”需求的自定义类别。
- 使用当前 YOLO11s 权重继续微调，而不是从通用预训练模型重新开始。
- 对真实数据和 COCO 子集保持合理比例，避免模型只记住单一家庭环境。

毕业设计价值：

这会让“智能家居物品找回”主题更贴近真实需求。尤其是“钥匙”目前并不在 20 类类别中，如果论文或演示中提到“找钥匙”，后续应将其纳入自定义类别。

### 5. 对小目标检测做专项优化

家庭物品中遥控器、鼠标、手机、书本等目标尺寸较小，容易因为输入分辨率、遮挡或光照导致漏检。

建议：

- 训练时尝试 `imgsz=800` 或 `imgsz=960`，与当前 `imgsz=640` 做对比。
- 在推理端允许用户选择“速度优先”和“精度优先”模式。
- 对小物体类别单独分析 AP，如果某几类明显偏低，再有针对性地补样本。
- 在客户端推流模式下可提供更高分辨率选项，但需要结合服务器性能做限流。

权衡：

更高分辨率会提升小目标效果，但会降低 FPS、增加云服务器负载。建议用实验表格记录不同分辨率下的 FPS 和 mAP，形成工程取舍。

### 6. 建立 YOLO11n 与 YOLO11s 的部署对比

当前仓库中同时存在 YOLO11n 和 YOLO11s 权重。YOLO11s 精度更好，但模型更大；YOLO11n 更轻量，适合 CPU 或低配服务器。

建议：

- 在同一段测试视频上对比 YOLO11n 与 YOLO11s 的 FPS、CPU/GPU 占用、mAP 或检测稳定性。
- 在后端配置中加入模型选择项，例如 `MODEL_VARIANT=yolo11n/yolo11s`。
- 若服务器没有独立 GPU，建议优先考虑 YOLO11n、ONNX Runtime、OpenVINO 或更低输入分辨率。
- 若服务器有 GPU，可继续使用 YOLO11s，并尝试半精度 FP16 或 TensorRT 导出。

毕业设计价值：

这可以形成“模型轻量化与部署性能分析”小节，体现算法效果和工程部署之间的平衡。

### 7. 优化 WebSocket 推流链路

当前客户端推流模式每 200ms 发送一帧，约 5 FPS，分辨率固定为 640x480，JPEG 质量为 0.75。这个设计简单稳定，但公网环境下仍可能受网络抖动、Cloudflare 代理策略、服务器性能影响。

建议：

- 增加自适应帧率：当服务器返回变慢时，自动降低推流 FPS。
- 增加前端队列丢帧机制：如果上一帧还未返回，不继续堆积新帧。
- 后端记录每帧处理耗时，并返回给前端显示。
- 在 WebSocket 接口限制最大帧大小，避免过大图片拖垮推理服务。
- 为本地视频推流增加暂停、单帧检测、倍速控制，便于演示和调试。

毕业设计价值：

这部分能体现系统对“公网云端推理”场景的适配，而不只是本地 demo。

### 8. 增加多摄像头和房间区域概念

当前数据库表中已有 `camera_id` 字段，但 `object_last_seen` 对 `object_class` 做唯一约束，意味着同一种物品在多摄像头场景下只能保存一个最后位置，不适合多个房间同时运行。

建议：

- 将 `object_last_seen` 的唯一约束从 `object_class` 调整为 `(camera_id, object_class)`。
- 前端增加摄像头管理页：名称、位置、RTSP 地址、启停状态。
- 数据库中增加 `rooms` 或 `camera_locations` 表，用于记录“客厅、卧室、书房”等语义位置。
- 检索结果从“最后出现在画面中的坐标”升级为“最后出现在某个房间、某个摄像头、画面哪个区域”。

毕业设计价值：

这会显著增强智能家居属性，让系统从单路视频检测扩展为家庭空间记忆系统。

### 9. 增加快照生命周期管理

当前快照保存在 `backend/snapshots/`，并通过 volume 持久化。如果系统长时间运行，快照数量会持续增加，占用磁盘空间。

建议：

- 增加快照保留策略，例如只保留最近 7 天或最近 5000 张。
- 增加定时清理任务，清理数据库中不再引用的快照。
- 前端统计页面显示快照占用空间。
- 对重要检索结果提供“收藏/锁定”功能，避免被自动清理。

毕业设计价值：

这属于长期运行可靠性优化，容易实现，也容易在论文中解释。

### 10. 引入数据库迁移工具

当前后端使用 SQLAlchemy `Base.metadata.create_all()` 自动建表，适合首次初始化，但后续如果修改表结构，例如支持多摄像头唯一约束、用户登录、房间表，就不方便平滑升级。

建议：

- 引入 Alembic 管理数据库迁移。
- 每次表结构变更都生成迁移脚本。
- 部署时自动执行迁移，避免手工改表。

毕业设计价值：

这可以让项目具备更成熟的工程维护能力。

### 11. 增加运行监控和健康检查

当前后端已经有 `/api/health`，但 Docker Compose 里还没有容器级 healthcheck，也缺少日志轮转和资源监控。

建议：

- 在 `docker-compose.yml` 中为前后端增加 `healthcheck`。
- 后端输出结构化日志，记录推理耗时、FPS、WebSocket 连接数、数据库写入耗时。
- 对容器日志设置轮转，避免日志占满磁盘。
- 增加简单的监控面板或 Prometheus 指标，如 `inference_fps`、`db_write_count`、`ws_connections`。

毕业设计价值：

这部分可体现部署运维完整性，尤其适合已经上云的项目。

### 12. 优化前端构建体积

本地执行前端生产构建可以通过，但 Vite 提示部分 chunk 超过 500 KB。其中统计页引入 ECharts、Element Plus 等依赖后体积较大。

建议：

- 对 ECharts 组件做按需引入，只加载柱状图、折线图、饼图需要的模块。
- 对 Element Plus 做自动按需导入，减少首屏包体。
- 将统计页、检索页继续保持路由级懒加载。
- 配置 `manualChunks` 或 Vite/Rolldown 代码拆分，让 `echarts`、`element-plus`、`vue-router`、`axios` 分包缓存。
- 对 CDN 场景设置合理缓存头，静态资源长缓存，HTML 不强缓存。

毕业设计价值：

这能和 Cloudflare CDN 部署结合起来，形成“前端性能优化与缓存策略”的说明。

### 13. 增强检索体验

当前检索以类别为中心，例如选择 `laptop` 后查询最后出现位置。用户体验上还可以更贴近自然语言和家庭使用场景。

建议：

- 增加中文关键词搜索，如“笔记本”“手机”“遥控器”直接匹配类别。
- 增加“最近出现”“最常出现”“今天出现过”的快捷筛选。
- 在快照预览上叠加检测框坐标，突出目标位置。
- 增加“物品时间线”，展示同一物品在一天内多次出现的快照。
- 如果后续扩展多摄像头，可显示“客厅摄像头左下角”这种更友好的空间描述。

毕业设计价值：

这部分能让系统从技术演示进一步变成“可用的找物工具”。

### 14. 增加自动化验证和 CI/CD

当前可以本地执行：

```text
python -m compileall backend
npm run build
```

建议：

- 添加 GitHub Actions，在每次提交时自动运行后端语法检查和前端构建。
- 添加 Docker 镜像构建验证，确保 README 中的一键部署命令始终可用。
- 对依赖做基础安全扫描。
- 给生产部署增加版本号，例如镜像 tag 对应 Git commit。

毕业设计价值：

这可以提升项目的规范性，也能避免答辩前临时改动导致部署不可用。

### 15. 补充部署文档中的 IPv6-only 与 Cloudflare 说明

当前项目已经实际部署在 IPv6-only 德国云服务器上，并使用 WARP 获得 IPv4 出口，这是一个很好的工程亮点。建议把这部分写进 README 或单独部署文档。

建议补充内容：

- IPv6-only 服务器为什么需要 WARP。
- Docker 构建阶段如何通过 WARP 访问 IPv4 资源，例如 pip、npm、apt 源。
- Cloudflare CDN 对 WebSocket、MJPEG 流和 API 的代理注意事项。
- `/api/`、`/api/video/ws`、`/api/video/ws/push`、`/snapshots/` 的反向代理策略。
- SSL 对浏览器 `getUserMedia` 摄像头权限的必要性。

毕业设计价值：

这部分能体现项目部署不是停留在本地，而是真正考虑了云服务器网络限制和浏览器安全策略。

## 六、近期最推荐实施的 5 项

如果后续时间有限，建议按以下顺序优化：

1. 给公网系统增加访问控制，至少先用 Cloudflare Access 或 Nginx Basic Auth。
2. 补全模型验证脚本，生成每类指标、混淆矩阵和验证样例图。
3. 用真实家庭场景抽帧数据继续微调，优先补充小目标和“找物”强相关类别。
4. 增加快照清理机制，避免云服务器磁盘长期累积。
5. 把 IPv6-only、WARP、SSL、CDN、Docker 部署流程写成正式部署说明。

这 5 项的共同特点是：实施成本相对可控，同时对论文、答辩和真实部署都有直接帮助。

- 前端构建存在大 chunk 提示，建议后续做 ECharts 和 Element Plus 的按需加载与分包优化。

## 八、总结

当前 YOLO-SmartHome 已经完成从模型训练、后端推理、前端交互、数据库持久化到 Docker 云端部署的主链路，具备完整毕业设计作品形态。后续改进应优先围绕三条线展开：

1. 安全与稳定：公网访问控制、敏感配置保护、快照清理、健康检查。
2. 模型与数据：补全评估报告、真实场景微调、小目标优化、自定义找物类别。
3. 产品与部署：多摄像头、房间语义、检索体验、前端性能、IPv6-only 云服务器部署文档。

如果这些优化逐步补齐，项目可以从“毕业设计演示系统”进一步发展为一个更接近真实智能家居应用的视觉感知与物品检索平台。
