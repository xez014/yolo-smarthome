# 基于 YOLO 的智能家居视觉感知与历史物品检索系统

## 摘要

随着智能摄像头、物联网终端与家庭宽带环境的普及，家庭视频数据正在从“被动记录”逐步转向“主动理解”。传统家居监控系统通常只能提供实时预览和录像回放，缺乏对画面中物品类别、位置和出现时间的结构化表达，难以支撑“物品最后出现在哪里”等面向日常生活的检索需求。针对这一问题，本文设计并实现了一套基于 YOLO 的智能家居视觉感知与历史物品检索系统。系统以 YOLO11 目标检测模型为核心，构建面向家居场景的 SmartHome-COCO 数据集，选取 COCO 2017 中 20 类常见室内物体进行迁移训练；后端采用 FastAPI、OpenCV、Ultralytics 与 SQLAlchemy 构建推理服务和数据持久化模块；前端采用 Vue 3、Element Plus 与 ECharts 构建实时监控、历史检索、统计分析和初始化配置界面。

在视频接入方面，系统同时支持服务端网络流拉取和浏览器客户端推流两种模式。服务端拉流模式通过 YOLO `track` 接口与 ByteTrack 跟踪器维护目标 ID，结合连续帧确认、数据库写入节流和快照间隔控制降低遮挡、光照变化和重复写库带来的影响；客户端推流模式利用浏览器 `getUserMedia` 采集本地摄像头或本地视频帧，并通过 WebSocket 将 JPEG 帧发送至云端服务完成无状态单帧推理。实验结果表明，当前部署使用的 YOLO11s 权重在训练日志最后一轮达到 Precision 0.67572、Recall 0.61442、mAP@0.5 0.65508、mAP@0.5:0.95 0.48428，能够满足毕业设计阶段对实时物品检测、历史位置记录和可视化检索的功能验证需求。本文的主要工作在于将轻量级目标检测、多目标跟踪、浏览器推流、关系型数据库检索与前后端可视化系统集成为一个可部署的智能家居视觉记忆平台，为家庭物品数字化管理提供了一种工程实现方案。

**关键词：** 智能家居；目标检测；YOLO11；ByteTrack；WebSocket；历史物品检索

## Abstract

With the widespread adoption of smart cameras, Internet of Things devices, and home broadband networks, household video data is evolving from passive recording toward active semantic understanding. Conventional home surveillance systems mainly provide live preview and video playback, but they rarely transform object categories, spatial positions, and timestamps into structured data. As a result, they cannot effectively answer practical questions such as “where was this object last seen?”. To address this problem, this thesis designs and implements a YOLO-based smart home visual perception and historical object retrieval system. The system uses YOLO11 as the core object detection model and constructs a SmartHome-COCO dataset by extracting 20 common indoor object categories from COCO 2017. The backend is built with FastAPI, OpenCV, Ultralytics, and SQLAlchemy for model inference, video processing, and data persistence, while the frontend is implemented with Vue 3, Element Plus, and ECharts for real-time monitoring, object retrieval, statistical visualization, and system setup.

For video access, the system supports both server-side stream pulling and browser-side stream pushing. In server-side pulling mode, the YOLO `track` interface and ByteTrack are used to maintain object identities across frames, while persistence rules such as consecutive-frame confirmation, database write throttling, and snapshot intervals are introduced to reduce the impact of occlusion, lighting changes, and duplicate writes. In browser-side pushing mode, `getUserMedia` captures local camera or local video frames, and WebSocket transmits JPEG frames to the cloud backend for stateless single-frame inference. According to the training log, the deployed YOLO11s model reaches a precision of 0.67572, recall of 0.61442, mAP@0.5 of 0.65508, and mAP@0.5:0.95 of 0.48428 in the final recorded epoch. These results indicate that the system can support real-time object detection, historical location recording, and visual retrieval in the scope of this graduation project. The main contribution of this work is the integration of lightweight object detection, multi-object tracking, browser-based streaming, relational database retrieval, and full-stack visualization into a deployable smart home visual memory platform.

**Keywords:** smart home; object detection; YOLO11; ByteTrack; WebSocket; historical object retrieval

## 1 绪论

### 1.1 研究背景与意义

智能家居系统的发展经历了从单点设备控制到多设备联动、再到智能感知的演进过程。摄像头作为家庭环境中信息密度较高的传感器，能够同时捕获空间布局、人员活动和物品状态。然而，普通监控系统大多只保存连续视频或短片段事件，用户在查找某个物品时仍需要手动回放大量视频，效率较低。若能够将视频中的物品类别、出现时间、空间位置和截帧快照转化为结构化数据，监控系统即可从“录像工具”升级为“视觉记忆工具”。

目标检测技术为上述能力提供了算法基础。Faster R-CNN 通过区域建议网络提升了两阶段检测器的效率，但其完整流程仍更适合精度优先的离线检测场景。YOLO 将目标检测表述为单阶段回归问题，使模型能够在一次前向传播中直接输出边界框与类别概率，显著提升了实时性。近年来 Ultralytics YOLOv8 与 YOLO11 进一步在模型结构、任务接口和部署工具链上完成工程化整合，使中小规模项目能够以较低成本完成训练、验证、推理与跟踪部署<sup>[1-2]</sup>。

本课题面向家庭物品“看见、记住、可检索”的应用需求，设计并实现基于 YOLO 的智能家居视觉感知与历史物品检索系统。该系统既关注检测模型本身的有效性，也关注模型与视频流、数据库、Web 前端和部署环境之间的工程闭环，具有明确的实用价值和毕业设计研究意义。

### 1.2 国内外研究现状

从检测算法看，早期两阶段检测器通常具有较高精度，但推理速度难以满足连续视频流场景。YOLO 系列模型以端到端、单阶段、实时检测为主要特征，在安防监控、机器人、室内导航和智能零售中被广泛采用。YOLOv8 官方文档指出，该版本采用 anchor-free split head，并在精度与速度之间提供多尺度模型选择<sup>[1]</sup>；YOLO11 系列在 COCO 检测任务上提供 n、s、m、l、x 等不同规模，其中 YOLO11n 约 2.6M 参数，YOLO11s 约 9.4M 参数，适合在轻量部署与精度需求之间进行权衡<sup>[2]</sup>。

从室内场景看，家居环境具有光照变化、遮挡频繁、目标尺度差异大、小物体易漏检等特点。Zhao 等人在智能室内购物场景中改进 YOLOv8n，引入 C2f-ParNet、CARAFE 与 WIoUv3，以提升复杂室内环境下小中型物体和遮挡物体的检测能力<sup>[3]</sup>。Pokuciński 与 Mrozek 研究了 YOLOv5 在室内等距矩形全景图像中的检测适应性，指出室内全景图像虽然包含丰富目标信息，但畸变、高分辨率和光照不连续会增加检测难度<sup>[4]</sup>。这些研究说明，室内视觉感知不只是模型套用问题，还需要结合采集视角、图像质量与系统部署方式进行整体设计。

国内高校的相关研究也为本文系统设计提供了参考。电子科技大学郭磊等针对小目标检测问题改进 YOLOv5，强调数据增强、特征表达与小目标召回能力的提升<sup>[9]</sup>；四川大学章程军等在车辆检测任务中引入 OSA 模块、非局部注意力机制和加权非极大值抑制，验证了特征增强和框筛选策略在复杂光照与密集目标场景中的有效性<sup>[10]</sup>；吉林大学赵畅的硕士论文围绕复杂场景下 YOLOv5 人脸检测展开网络结构和参数优化研究<sup>[11]</sup>；电子科技大学王选的硕士论文则从特征融合、注意力机制和复杂环境目标检测角度分析了小尺度、聚集和图像质量退化等问题<sup>[12]</sup>。上述中文文献虽然应用场景各有差异，但其关于轻量化检测、小目标增强、注意力机制与复杂环境鲁棒性的讨论，与本文面向家居小物品和遮挡场景的技术路线具有较强关联。

从目标跟踪看，单帧检测只能回答“当前帧中有什么”，无法稳定回答“同一物品是否连续出现”。Deep SORT 在 SORT 基础上加入外观特征度量，能够缓解遮挡带来的身份切换，但也引入额外特征提取开销。ByteTrack 则通过关联高、低置信度检测框恢复被遮挡或低分目标轨迹，在 MOT17 上取得 80.3 MOTA、77.3 IDF1，并保持接近实时的处理速度<sup>[5]</sup>。Ultralytics 跟踪文档也提供了 `bytetrack.yaml` 作为内置跟踪器配置，便于与 YOLO 检测模型直接结合<sup>[6]</sup>。

从 Web 流媒体工程看，服务端直接拉取 RTSP/RTMP/HTTP 视频流适合局域网或公网可达摄像头，但在家庭网络中常受 NAT 和设备权限限制。浏览器端 `getUserMedia` 能够请求用户授权并获取摄像头 `MediaStream`，但该 API 在现代浏览器中要求安全上下文，即 HTTPS 或 localhost<sup>[7]</sup>。WebSocket 则提供浏览器与服务端之间的长连接通信能力，可发送文本或二进制数据，适合将视频帧以 JPEG 字节流方式传输到推理服务<sup>[8]</sup>。因此，服务端拉流与浏览器推流的双模式设计能够覆盖更多实际部署场景。

### 1.3 本文主要工作

本文围绕“智能家居视觉记忆”目标完成以下工作：

1. 构建 SmartHome-COCO 数据集处理流程，从 COCO 2017 中提取 20 类常见室内物品并转换为 YOLO 训练格式。

5. 设计 Vue 3 前端大屏，实现实时检测展示、历史物品检索、中文关键词查询、统计图表、登录鉴权和系统初始化配置。
6. 使用 Docker Compose 组织前端、后端服务，并加入 JWT 鉴权、快照生命周期管理和健康检查等工程化能力。

## 2 相关技术基础

### 2.1 YOLO 目标检测算法

目标检测任务需要同时完成类别判别与边界框定位。Faster R-CNN 等两阶段方法先生成候选区域，再对候选区域分类和回归，能够取得较高精度，但流程较长。YOLO 将检测任务转化为单阶段回归问题，由单个神经网络直接从整幅图像预测边界框和类别概率，因而具有天然的实时性优势。

本文项目采用 Ultralytics YOLO11 体系。YOLO11 提供多种模型规模，能够在不同算力环境下选择部署方案<sup>[2]</sup>。项目中同时保留 YOLO11n 与 YOLO11s 两类训练权重，其中生产配置实际加载 `results-01/runs/train_smarthome_s/weights/best.pt`，即 YOLO11s 权重。相较 YOLO11n，YOLO11s 参数量和权重文件更大，但通常具有更强的特征表达能力，适合毕业设计演示阶段对检测稳定性的需求。

YOLO11 检测模型可从工程应用角度抽象为输入预处理、骨干特征提取、颈部特征融合、检测头输出和后处理五个部分。输入图像首先被缩放至统一尺寸并归一化，随后通过骨干网络提取多尺度语义特征；颈部网络融合高、低层特征以兼顾大目标语义和小目标细节；检测头输出边界框坐标、类别概率和置信度，最后经 NMS 或跟踪器关联得到最终检测结果。图 2-1 为本文采用的 YOLO11 模型抽象结构示意图，该图用于说明系统中的模型调用逻辑，并非官方逐层网络结构图。

```mermaid
flowchart LR
    A[输入图像/视频帧<br/>640x640] --> B[预处理<br/>Resize/Normalize]
    B --> C[Backbone<br/>多尺度特征提取]
    C --> C1[浅层纹理特征<br/>小目标细节]
    C --> C2[深层语义特征<br/>类别判别]
    C1 --> D[Neck<br/>FPN/PAN 特征融合]
    C2 --> D
    D --> E[Detection Head<br/>bbox/class/conf]
    E --> F[后处理<br/>NMS/阈值过滤]
    F --> G[检测结果<br/>类别 + 置信度 + 边界框]
    G --> H[ByteTrack/业务持久化]
```

### 2.2 多目标跟踪与 ByteTrack

在视频场景中，同一物品可能连续出现在多个帧中。若将每一帧检测结果都直接写入数据库，会产生大量重复记录；若只做单帧检测，又无法判断检测目标是否稳定出现。ByteTrack 的核心思想是关联每个检测框，而不只保留高置信度检测框。其先使用高置信度检测框进行第一轮匹配，再利用低置信度检测框恢复被遮挡或模糊的轨迹，从而减少轨迹碎片化<sup>[5]</sup>。

本系统在服务端拉流模式下调用 `model.track(..., tracker="bytetrack.yaml", persist=True)`，使检测结果带有 `track_id`。在应用层，系统进一步维护 `track_id -> 连续出现帧数` 的内存计数，只有当目标连续出现帧数达到阈值后才进入持久化流程。该策略将算法层跟踪结果与业务层去抖规则结合，能够降低单帧误检和短暂遮挡对历史记录的干扰。

### 2.3 WebSocket 推流与浏览器媒体采集

家庭摄像头往往处于内网环境，公网服务器直接拉取 RTSP 流存在网络可达性问题。为增强系统适配性，本文设计了浏览器客户端推流模式。前端通过 `navigator.mediaDevices.getUserMedia` 请求摄像头权限，或者读取本地视频文件，以固定分辨率将视频帧绘制到 Canvas，再使用 `canvas.toBlob` 编码为 JPEG。随后前端通过 WebSocket 将二进制帧发送到后端 `/api/video/ws/push` 接口，后端完成单帧 YOLO 推理后返回带检测框的 Base64 图像与结构化检测列表。

该方案的优点是无需摄像头暴露公网地址，且能够直接利用用户浏览器的设备权限；不足是每帧推理相互独立，无法天然维护跨帧 TrackID。因此，系统在推流模式下采用写库节流和快照间隔策略，以控制数据库压力和磁盘占用。

## 3 系统需求分析

### 3.1 功能需求

系统面向智能家居场景，需要实现以下核心功能：

1. 实时物品检测：识别家庭画面中的椅子、沙发、电视、笔记本电脑、鼠标、遥控器、手机、书籍等 20 类物品。
2. 视频接入：支持网络流地址输入，同时支持浏览器本地摄像头和本地视频文件推流。
3. 历史记录：将检测类别、置信度、归一化边界框、时间戳、摄像头标识和快照路径写入数据库。
4. 最后出现位置：维护每类物品最近一次出现的位置和快照，支持快速查询。
5. 统计分析：按日期、小时和类别聚合检测次数，并以图表展示。
6. 系统初始化：首次部署后可通过图形化页面配置 MySQL 连接并创建数据表。
7. 安全访问：通过管理员密码登录和 JWT Token 保护关键接口及 WebSocket 推流接口。

### 3.2 非功能需求

系统还需要满足以下非功能需求：

1. 实时性：视频流推理、检测结果返回和前端渲染需要保持可交互体验。
2. 可部署性：前后端、推理服务和持久化目录应能通过 Docker Compose 部署。
3. 可维护性：配置、路由、数据库模型、推理引擎和前端组件应保持模块化。
4. 数据安全性：数据库密码不应明文写入代码仓库，公网环境关键接口需要鉴权。
5. 存储可控性：检测快照应具备生命周期清理机制，防止长期运行导致磁盘占满。

### 3.4 业务流程分析

系统业务流程以“视频接入、模型推理、结构化存储、用户检索”为主线。管理员登录后可选择网络流、本地摄像头或本地视频作为输入源。网络流由后端主动拉取，浏览器本地源则由前端采集并通过 WebSocket 推送。后端统一完成 YOLO 推理、检测框绘制、数据清洗、快照保存和数据库写入，前端根据实时返回结果展示画面和检测列表。

## 4 系统总体设计

### 4.1 总体架构

系统采用前后端分离的 B/S 架构，整体分为数据层、算法层、后端服务层、前端展示层和部署层。

数据层由 SmartHome-COCO 数据集和 MySQL 数据库组成。数据集用于模型训练，数据库用于保存检测记录、最后出现位置和统计汇总。算法层由 YOLO11 目标检测模型、ByteTrack 跟踪器和 OpenCV 图像处理逻辑组成。后端服务层基于 FastAPI，提供认证、检测、视频、统计、初始化和快照访问接口。前端展示层基于 Vue 3，提供实时监控大屏、历史检索页、统计分析页、登录页和系统初始化页。部署层使用 Dockerfile 与 Docker Compose 组织服务，并通过 Nginx 托管前端静态资源及代理后端接口。

### 4.2 后端模块设计

后端主要模块如下：

1. `detection_engine.py`：系统核心推理引擎，负责模型加载、视频源打开、帧循环、YOLO 推理、ByteTrack 跟踪、检测框绘制、快照保存和数据库写入。
2. `models.py`：定义 `DetectionRecord`、`ObjectLastSeen` 和 `DetectionStat` 三张核心数据表。
3. `routers/video.py`：提供 MJPEG 视频流、实时检测 WebSocket、客户端推流 WebSocket、启动检测和停止检测接口。
4. `routers/detection.py`：提供类别列表、中文关键词检索、最后出现位置查询和历史记录分页查询接口。
5. `routers/stats.py`：提供概览统计、类别分布和时间分布接口。
6. `auth.py`：实现单管理员登录、JWT 签发和 HTTP/WebSocket 鉴权。
7. `snapshot_cleaner.py`：后台定时清理超过保留天数的快照文件。

## 5 数据集构建与模型训练

### 5.1 SmartHome-COCO 数据集构建

COCO 数据集包含复杂日常场景中的常见物体，并提供实例级标注，适合用于目标检测模型训练和迁移学习。本文项目基于 COCO 2017 构建 SmartHome-COCO 子集，选择与智能家居场景关系较强的 20 类物品：

`chair`、`couch`、`potted plant`、`bed`、`dining table`、`toilet`、`tv`、`laptop`、`mouse`、`remote`、`keyboard`、`cell phone`、`microwave`、`oven`、`toaster`、`sink`、`refrigerator`、`book`、`clock`、`vase`。

数据处理脚本 `data_pipeline.py` 使用 `pycocotools` 读取 COCO 标注文件，根据目标类别筛选图像和标注，并将 COCO `[x_min, y_min, width, height]` 边界框转换为 YOLO 归一化格式 `[class_id, center_x, center_y, width, height]`。最终生成符合 Ultralytics 训练要求的 `images/train`、`images/val`、`labels/train` 和 `labels/val` 目录结构，并由 `smarthome.yaml` 指定训练集、验证集和类别名称。

### 5.2 模型训练配置

训练脚本基于 Ultralytics `YOLO` Python API 实现。初始脚本以 `yolo11n.pt` 预训练权重为起点，设置 `epochs=100`、`imgsz=640`、`batch=32`、`device=0` 和自动混合精度训练。当前生产推理实际采用 `results-01/runs/train_smarthome_s/weights/best.pt`，即 YOLO11s 训练产物。后端 `config.py` 通过 `WEIGHTS_PATH` 指定该权重路径，并设置置信度阈值 `0.25`、NMS IoU 阈值 `0.5` 和输入尺寸 `640`。

本文进一步运行 `model_training/benchmark.py` 对 YOLO11n 与 YOLO11s 进行同验证集对比。对比表保存于 [docs/pics/yolo11n_vs_yolo11s_benchmark.md](pics/yolo11n_vs_yolo11s_benchmark.md)，结果如下：

| 指标 | YOLO11n | YOLO11s |
| --- | ---: | ---: |
| 参数量 | 2.59M | 9.44M |
| 文件大小 | 15.2 MB | 54.4 MB |
| Precision | 0.6964 | 0.6775 |
| Recall | 0.5230 | 0.6144 |
| mAP@0.5 | 0.6013 | 0.6552 |
| mAP@0.5:0.95 | 0.4384 | 0.4849 |
| 评估耗时 | 109.7s | 275.7s |

对比结果表明，YOLO11n 参数量和文件大小显著小于 YOLO11s，CPU 验证耗时也更低，适合低算力或速度优先场景；YOLO11s 在 Recall、mAP@0.5 和 mAP@0.5:0.95 上均高于 YOLO11n，更适合作为当前系统的生产推理模型。考虑到本系统更关注历史检索场景下的漏检控制，生产配置采用 YOLO11s 权重是较合理的工程取舍。需要说明的是，上述耗时是在 CPU 环境下得到，若部署 GPU、TensorRT 或 ONNX Runtime，绝对耗时会发生变化，但同一硬件环境下的相对比较仍具有参考价值。

## 6 系统实现

### 6.1 服务端拉流检测流程

服务端拉流模式由 `DetectionEngine.start()` 启动。系统根据输入源类型判断其为网络流或视频文件，再使用 OpenCV `VideoCapture` 打开视频源。推理循环中，系统读取每一帧并调用：

```python
results = self.model.track(
    frame,
    conf=config.CONFIDENCE_THRESHOLD,
    iou=config.IOU_THRESHOLD,
    imgsz=config.INPUT_SIZE,
    tracker=config.TRACKER_TYPE,
    persist=True,
    verbose=False
)
```

随后 `_parse_results()` 将模型输出转换为前端和数据库可直接使用的结构，包括类别 ID、英文类别、中文类别、置信度、像素坐标、归一化中心点与宽高、`track_id` 等字段。`_draw_detections()` 在帧上绘制边界框、中文标签、置信度和 FPS，最终供 MJPEG 流和快照保存使用。

持久化逻辑采用“跟踪稳定性 + 写库节流 + 快照间隔”的组合策略。系统维护每个 `track_id` 的连续出现计数，当目标短暂消失时逐步扣减而不是立刻删除，以容忍单帧漏检。只有达到 `TRACK_PERSIST_FRAMES` 的目标才写入数据库；同时通过 `DB_WRITE_INTERVAL` 控制写库最小间隔，通过 `SNAPSHOT_INTERVAL` 控制快照保存最小间隔。数据库写入在后台线程中执行，避免阻塞推理主循环。

模型推理结果到业务数据的转换流程如图 6-1 所示。YOLO 输出的原始张量并不直接写入数据库，而是经过类别映射、坐标归一化、中文名称补全、跟踪 ID 读取、稳定性判断和节流控制后，才转化为可供前端展示与历史检索使用的业务字段。

```mermaid
flowchart LR
    A[视频帧 frame] --> B[YOLO11 track/predict]
    B --> C[boxes.cls<br/>boxes.conf<br/>boxes.xyxy<br/>boxes.id]
    C --> D[类别编号转英文类名]
    D --> E[英文类名转中文显示名]
    C --> F[xyxy 像素框]
    F --> G[归一化 bbox_x/y/w/h]
    C --> H[读取 track_id]
    E --> I[DetectionRecord 字段]
    G --> I
    H --> I
    I --> J{达到稳定帧数和写库间隔}
    J -- 否 --> K[仅返回实时检测结果]
    J -- 是 --> L[保存快照]
    L --> M[写入 detection_records]
    M --> N[更新 object_last_seen]
    M --> O[更新 detection_stats]
```

### 6.2 客户端推流检测流程

客户端推流模式由 `LocalStreamer.vue` 实现。启动摄像头时，前端调用 `navigator.mediaDevices.getUserMedia({ video: { width: 640, height: 480 } })` 获取本地视频流；启动本地视频模式时，前端读取用户选择的视频文件并用 `<video>` 元素播放。采集定时器每 200 ms 将视频画面绘制到 640×480 的 Canvas，并以 JPEG 质量 0.75 编码为 Blob，然后通过 WebSocket 二进制消息发送至后端。

后端 `/api/video/ws/push` 接口首先通过 query 参数中的 JWT Token 完成鉴权，然后接收 JPEG 字节并调用 `infer_single_frame()`。该方法使用 `cv2.imdecode` 解码图像，调用 `model.predict()` 完成单帧推理，绘制检测框后返回 Base64 图像与检测结果列表。由于推流模式没有 TrackID，系统通过统一的写库节流和快照间隔策略降低高频重复数据。

### 6.3 历史检索与统计实现

历史检索模块围绕 `object_last_seen` 和 `detection_records` 两张表实现。用户可按英文类别或中文关键词查询物品最后出现位置。例如输入“笔记本”时，后端会在中文映射表中匹配 `laptop`，再查询该类别在 `object_last_seen` 中的最新记录。历史记录接口支持按类别、起止时间和分页参数查询 `detection_records`，便于展示某类物品在一段时间内的出现轨迹。

统计模块通过 `detection_stats` 表维护小时级聚合数据。每次写入检测记录时，后端同步更新当前日期、当前小时和当前类别对应的检测次数。前端统计页以图表方式展示类别分布和时间趋势，使用户能够了解家中物品在不同时间段的出现频率。

### 6.4 安全与部署实现

系统加入单管理员 JWT 鉴权。登录接口校验环境变量 `ADMIN_PASSWORD` 指定的管理员密码并签发 Token，后续关键 HTTP 接口通过 `Authorization: Bearer <token>` 访问，WebSocket 推流接口通过 query 参数传递 Token 并调用 `verify_ws_token()` 验证。数据库密码处理方面，系统优先读取 `DB_PASSWORD` 环境变量，初始化页面输入的密码只保存在运行时内存中，避免继续向 JSON 文件明文写入敏感信息。

部署方面，系统使用 `Dockerfile.backend`、`Dockerfile.frontend` 和 `docker-compose.yml`。后端容器挂载 `backend/data` 和 `backend/snapshots` 以持久化配置和快照，前端容器由 Nginx 托管并监听 `10080` 端口。Compose 文件为前后端配置 healthcheck，后端通过访问 `/api/health` 检查服务健康，前端通过 `wget --spider` 检查 Nginx 静态服务可用性。

### 7.3 模型效果分析

当前 YOLO11s 模型的总体指标达到毕业设计演示要求，但仍存在以下特点：

第一，Precision 高于 Recall，说明模型对已检出目标的判断相对可靠，但仍可能漏检部分目标。家居环境中小目标、遮挡目标和低光照目标较多，召回率仍有继续提升空间。

第二，mAP@0.5:0.95 明显低于 mAP@0.5，说明严格定位条件下边界框质量仍需优化。后续可通过提高训练输入尺寸、补充真实家庭数据、增加小目标样本和类别均衡策略改善。

第三，当前数据集来自 COCO 子集，类别覆盖常见家具和电器，但缺少钥匙、钱包、眼镜、耳机、充电器等“找物”强相关类别。若系统面向真实家庭使用，应增加自定义类别并进行人工标注微调。

### 7.4 工程效果分析

从工程角度看，系统的价值不只在于模型推理，而在于形成完整闭环。服务端拉流模式适合固定摄像头和网络可达视频源，能够利用 ByteTrack 维护轨迹连续性；客户端推流模式适合公网部署和临时演示，能够绕开家庭摄像头内网穿透问题。两种模式结合，提高了系统的适用范围。

持久化策略方面，`DetectionRecord` 保存明细，`ObjectLastSeen` 支撑快速检索，`DetectionStat` 支撑统计分析，三类数据各自面向不同查询场景，避免了只依赖单表造成的查询复杂度上升。快照清理、鉴权和健康检查使系统从简单 demo 更接近可长期运行的 Web 应用。

### 7.5 不足与改进方向

当前系统仍存在以下不足：

1. 训练数据主要来自 COCO，真实家庭监控视角、低光照、运动模糊和遮挡样本不足。
2. 浏览器推流模式为无状态单帧推理，尚未维护跨帧 TrackID。
3. `object_last_seen` 当前以 `object_class` 作为唯一约束，尚不适合多摄像头同类物品同时存在的场景。
4. 快照预览尚可进一步叠加检测框坐标，以突出目标区域。
5. 前端 ECharts 与 Element Plus 可继续按需加载，降低生产构建体积。

后续可围绕真实家庭数据微调、小目标专项优化、多摄像头与房间语义建模、WebSocket 丢帧保护、推理耗时回传和自动化 CI/CD 等方向继续完善。

## 8 总结与展望

本文设计并实现了一套基于 YOLO 的智能家居视觉感知与历史物品检索系统。系统以 YOLO11s 为核心检测模型，基于 COCO 2017 构建 20 类家居物品数据集，结合 FastAPI、OpenCV、ByteTrack、MySQL、Vue 3 和 WebSocket 实现实时检测、历史位置记录、快照保存、中文检索、统计分析、登录鉴权和容器化部署。实验和功能验证表明，系统已具备完整的毕业设计作品形态，能够将家庭视频流中的物品转化为可查询的结构化时空数据。

本文工作的创新点主要体现在系统集成层面：一是将目标检测与历史物品检索结合，使监控画面从被动记录转向结构化记忆；二是同时支持服务端拉流与浏览器推流，提升系统对不同网络环境的适应性；三是将跟踪去抖、写库节流、快照生命周期管理和 JWT 鉴权纳入完整工程设计，增强系统可部署性和可维护性。

未来，系统可进一步引入真实家庭数据标注与增量训练，扩展钥匙、钱包、眼镜等更贴近“寻找物品”的类别；在算法层面可尝试更高输入分辨率、模型蒸馏、ONNX/TensorRT 加速与多目标长期重识别；在应用层面可扩展多摄像头、房间语义、物品时间线和自然语言检索，使系统从毕业设计演示平台进一步发展为更完整的智能家居视觉记忆服务。
